# ADR 002 — PostGIS as the Spatial Analysis Engine

**Status:** Accepted
**Date:** 2025-01
**Deciders:** Initial project setup

---

## Context

Felt Like It stores, queries, and exports geospatial features. The application needs to:

1. Store geometry (points, lines, polygons) for user-created and imported features.
2. Return geometry to the browser as GeoJSON for MapLibre rendering.
3. Compute bounding boxes for auto-fit-to-layer.
4. Validate and normalise imported GeoJSON/CSV to WGS84.
5. (Phase 2+) Perform spatial operations: buffer, intersect, clip, spatial join.

The primary question was whether to store geometry as raw JSON in PostgreSQL or to use PostGIS.

Candidates evaluated:

| Option | Geometry storage | Spatial ops | Notes |
|---|---|---|---|
| PostgreSQL + PostGIS | Native geometry type | Full ST_* function set | Requires PostGIS extension |
| PostgreSQL + jsonb | Raw GeoJSON text/jsonb | None (application-side) | No spatial indexing |
| SQLite + SpatiaLite | Native geometry type | Limited ST_* | Complicates Docker; no concurrent writes |
| MongoDB | GeoJSON native | `$geoNear`, `$geoWithin` | Different stack; weaker transactional guarantees |

---

## Decision

Use **PostgreSQL 16 + PostGIS 3.4** (`postgis/postgis:16-3.4-alpine` Docker image) as the sole database for both relational data (users, maps, layers) and spatial data (features).

---

## Rationale

### Unified datastore

Keeping spatial and relational data in one database eliminates cross-database joins, distributed transactions, and additional operational complexity. All foreign key relationships (users → maps → layers → features) are enforced by the RDBMS.

### PostGIS is the industry standard

PostGIS is the most widely deployed open-source spatial database extension. It implements the OGC Simple Features specification and supports the full ISO/OGC SQL/MM geometry type hierarchy. Knowledge is transferable; documentation is excellent; tooling support is broad (QGIS, GeoServer, Mapbox, Esri).

### ST_AsGeoJSON / ST_GeomFromGeoJSON

The application's read/write pattern is straightforward:

- **Write**: `ST_GeomFromGeoJSON($geojson_string)` — accepts standard GeoJSON geometry objects.
- **Read**: `ST_AsGeoJSON(geometry)` — returns a GeoJSON geometry string, consumed directly by MapLibre.

No WKB encoding/decoding is required in application code. No binary parsing library (`wkx`, `terraformer`) is needed. This reduces the dependency surface and keeps geometry transformation logic in the database.

### Spatial indexing

The `features.geometry` column carries a GIST index:

```sql
CREATE INDEX features_geometry_idx ON features USING GIST (geometry);
```

This enables efficient spatial queries (bounding box filter, nearest neighbour) that are not possible with jsonb storage.

### Phase 2+ spatial analysis

PostGIS natively supports the operations planned for Phase 4 (buffer, clip, intersect, union, dissolve, spatial join, choropleth aggregation). Implementing these in the application layer (e.g., with Turf.js) would require fetching all features to the server, processing them in Node.js, and writing results back — an order of magnitude slower than a single PostGIS query.

### SRID consistency

All geometries are stored in SRID 4326 (WGS84 geographic coordinates), which is the native coordinate system for GeoJSON and MapLibre. CRS normalisation for imported data (Phase 2: Shapefile/KML) happens at import time in `packages/geo-engine/src/transform.ts` before insertion, keeping the database uniform.

---

## Consequences

- The Docker Compose setup must use `postgis/postgis:16-3.4-alpine` rather than plain `postgres:16-alpine`. This image is larger (~120 MB vs ~80 MB) and changes more slowly.
- Drizzle ORM does not natively support PostGIS geometry types. A `customType` wrapper is used that delegates all spatial operations to raw SQL (`db.execute(sql\`...\`)`). This means geometry columns cannot be used in Drizzle's type-safe query builder — only in raw SQL strings.
- `drizzle-kit generate` does not understand `geometry(Geometry,4326)`. The initial migration (`0000_initial.sql`) is hand-authored rather than generated by Drizzle Kit, to correctly set up PostGIS extensions, the geometry column, GIST index, and `updated_at` triggers.
- Developers running the app locally must have Docker available to run PostGIS. There is no SQLite fallback.
